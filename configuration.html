Configuration guide

While I think the config file itself should be very self-explanatory, some settings might require some more explanation.

Since the configuration is just a big JSON object in a JavaScript file, the configuration can be defined in many more ways than just what the example looks like. The option names are the "path" in the object structure, e.g. the path "channels[i].name" could describe the container of "#foo" in the following structure:

var config = { channels: [ { name: "#foo" } ] };

channels[i].name

It's just a name, nothing more. Pick anything. For example, if your channel is called "#MyChannel" and it's in the IRC network "MyNetwork", you could, for example, name it "#MyChannel @ MyNetwork"

var config = { channels: [ { name: "#MyChannel @ MyNetwork" } ] };

channels[i].theme

This setting is case sensitive.

This changes how the generated statistics look like. Currently there's only one option, "default". If you create more, or if one day the package comes with more, you can insert the name of the theme folder under "themes/" here. For example, if in the JSIS installation directory you ran the command "ls themes" and got the result:

$ ls themes default cooltheme

... you could use "default" or "cooltheme" as the option.

var config = { channels: [ { theme: "cooltheme" } ] };

channels[i].destination

This setting is case sensitive.

The destination directory you want to put the generated HTML, CSS, and JavaScript in. If you're running JSIS on the server hosting the statistics, you probably want this to be under your web root directory. E.g., if your web root is /var/www/html/ and you want your stats to be accessible under http://mydomain/stats/ ... you could try setting this as "/var/www/html/stats/". var config = { channels: [ { theme: "/var/www/html/stats/" } ] }; NOTE: I will not help you to publish your generated stats if you run into issues.

channels[i].logFormat

This setting is case sensitive.

The format your log files are in, or really the name of the class that will do the parsing. The valid values can be figured out by executing "ls classes/LogReaders/" in your JSIS installation directory. For example, if you got the result:

$ ls classes/LogReaders/ eggdrop.js irssi.js

... the valid values would be "eggdrop" and "irssi", which obviously are for parsing logs from Eggdrop and Irssi.

For example, if your log files are from an Eggdrop bot, you'll want to enter "eggdrop", as in: var config = { channels: [ { logFormat: "eggdrop" } ] };

channels[i].logEncoding

This setting is case sensitive (afaik).

Wrong setting can produce broken output.

If your log files are not in UTF-8, we should really REALLY convert them to UTF-8 using iconv. If this setting is set to anything except "UTF-8", we will put them through an iconv conversion from the encoding entered here to UTF-8, prior to parsing them. List of iconv supported encodings

You can try and find out what type the file is with the "file" util. E.g. if one of your log files is at /home/lietu/logs/mylog.log, you could run:

$ file -bi /home/lietu/logs/mylog.log text/plain; charset=us-ascii

.. the output then quite clearly says that we should be using the ASCII encoding (in this example, your results might vary).

For example if your log files are using ASCII encoding, you can use:

var config = { channels: [ { logEncoding: "ASCII" } ] };

channels[i].logTimezone

The timezone your log files are in. You really only need this in case your log files are on a different timezone than the server you're generating the logs on.

Format is +|-HH:MM as offset from GMT. Set to null to use server timezone, which is also the default.

E.g. +03:00 or -07:00.

var config = { channels: [ { logTimezone: "+03:00" } ] };

channels[i].logPath

This setting is case sensitive.

The location your log files are in ... for safety, you should really use a full path here. For example, if your log files are stored in "/home/lietu/logs/", you should enter that, e.g.: var config = { channels: [ { logPath: "/home/lietu/logs/" } ] };

channels[i].logFileFilter

If your logPath contains logs for many channels, you can use this variable to define a regular expression that will be used to match only the files you want to include in this channel's stats. The regular expression will be matched against the full path to the file, and if you need it to be case insensitive, you will have to make it case insensitive yourself (/foo/i).

... for help with regular expressions, check:

  • http://www.visibone.com/regular-expressions/
  • http://www.cheatography.com/davechild/cheat-sheets/regular-expressions/

For example to match only files that start with "mychannel.log" in a case insensitive manner, you could use:

var config = { channels: [ { logFileFilter: /\/mychannel.log[^/]*/i } ] };

channels[i].recursive

If the logPath should be parsed recursively. That is, if we should look in subdirectories and subdirectories of subdirectories (etc.) of the logPath. If your log files are organized in folders by year, or similar, then you might want to enable this setting. There rarely are any side-effects of leaving it on when you don't need to. var config = { channels: [ { recursive: true } ] };

Back to table of contents

channels[i].maxLogFiles

If you want to limit the amount of log files to parse, set the limit here. JavaScript has a magic constant called "Infinity" that you can use to parse them all. Good values could be: 7, 14, 30, 90, 180, 365, Infinity

For example to always generate stats for the last two weeks, you could use:

var config = { channels: [ { maxLogFiles: 14 } ] };

channels[i].statsTimezone

If you want to adjust the stats to be rendered in a different timezone than the server is at, then set to the timezone you want it to be rendered in. If you are going to use this, you probably want to make sure logTimezone is correct.

Format is +|-HH:MM as offset from GMT. Set to null to use server timezone, which is also the default. For GMT, use +00:00.

E.g. +03:00 or -07:00.

var config = { channels: [ { statsTimezone: '-07:00' } ] };

Back to table of contents

channels[i].statsTimezoneText

What to show as being the timezone when outputting to the stats. By default it is "GMT+|-HH:MM", e.g. "GMT-07:00", so if you want to replace this with something like "DST", you can set the text to be used instead here.

var config = { channels: [ { statsTimezoneText: 'GMT (Greenwich Mean Time)' } ] };

channels[i].widgets

This setting is for customizing the contents and the order of items on your stats page. If you define this list, the page will only include the widgets listed here, in the order they are listed. You can see the available widgets by running "ls -1d widgets/*/" in your JSIS installation folder.

For example, if the output was:

$ ls -1d widgets/*/ widgets/BigNumbers/ widgets/DailyActivity/ widgets/Example/ widgets/LatestTopics/ widgets/MostActiveNicks/ widgets/MostActiveNicksByHour/ widgets/MostActiveTimes/ widgets/MostReferencedNicks/ widgets/MostReferencedURLs/ widgets/MostUsedWords/ widgets/OtherInterestingNumbers/ widgets/Smileys/ widgets/UsersWithMostNicknames/

The valid options would be:

  • BigNumbers
  • DailyActivity
  • Example (this is an example widget with no real use)
  • LatestTopics
  • MostActiveNicks
  • MostActiveNicksByHour
  • MostActiveTimes
  • MostReferencedNicks
  • MostReferencedURLs
  • MostUsedWords
  • OtherInterestingNumbers
  • Smileys
  • UsersWithMostNicknames

For example, if you would only want your stats to contain the list of most active nicks, and latest topics, you could use:

var config = { channels: [ { widgets: ['MostActiveNicks', 'LatestTopics'] } ] };

channels[i].customHeadHtml

If you want to easily add your own HTML code to the section of the HTML, you can insert it here. For example, if you want Google Analytics for your stats, just get the HTML code from Google and paste it here. You can do a lot of other things too with it, if you're creative.

For example, to always show an annoying "Welcome to the stats" message for every page load, you could use this (please don't):

var config = { channels: [ { customHeadHtml: '<script type="text/javascript">alert("Welcome to the stats");</script>' } ] };

Please note that if you don't escape the quotes properly, you can easily break the configuration file.

channels[i].soloLength

If you want to tune how many lines/actions in a row does one one need to say before it's considered "soloing", you can do it with this.

For example, to consider 4 consecutive lines/actions from one person "soloing", you can use:

var config = { channels: [ { soloLength: 4 } ] };

channels[i].wordMinLength

Minimum length of words to be shown in the "most used words" widget. Easy way to remove all the plain useless words like "and", "or", etc.

var config = { channels: [ { wordMinLength: 4 } ] };

channels[i].ignoreWords

Space separated list of words to be ignored for the "most used words" widget. If you set this, the defaults will not be used at all. If you want to add words, please contribute to the project instead.

For example, in a Finnish channel you might want to replace the default list with common Finnish pronouns, etc., e.g.:

var config = { channels: [ { foulWords: 'ja ei tai'] } ] };

You might end up with character set issues with non-ASCII characters. Good luck resolving those, if you bump into any.

channels[i].userConfig

User configuration. This consists of several per-user sections.

Aliases

JavaScript array of aliases for this user. They can be space separated strings with simple wildcards (* and ?), or JavaScript RegExp objects, that will be tested against the nickname after it's been converted to lowercase characters.

For example, to give te user "lietu" the aliases "lietu-", "*lietu" and do a RegEx alias /^lie.*tu$/, you could do the following:

var config = { channels: [ { userConfig: { lietu: { aliases: ["lietu- *lietu", /^liet.*tu$/] } } } ] };

... for help with regular expressions, check:

log

This is the logging configuration. The application is logging a lot of things, with different log levels, so you can get a varying level of output depending on your level of interest in what's going on.

  • DEBUG: Debugging information, mainly useful for developers.
  • INFO: Information on what's going on, e.g. if you want to know how long it takes to parse some files.
  • MESSAGE: Important messages.
  • WARNING: If the application finds something wrong, that will not cause a critical failure, it will issue a warning.
  • ERROR: Some severe errors might end up here.
  • CRITICAL: Seriously bad errors end up here.

Generally you'll probably want to enable MESSAGE, WARNING, ERROR and CRITICAL. Maybe when automating generation you might want to only enable ERROR and CRITICAL. You can just leave out the ones you don't want.

You can also define separate destinations for every log level, either with filenames, or the two magic strings STDOUT and STDERR.

For example to log and dump ERROR and CRITICAL levels to STDERR, you could use the following:

var config = { log: { 'ERROR': ['STDERR', 'errors.log'], 'CRITICAL': ['STDERR', 'errors.log'] } };
